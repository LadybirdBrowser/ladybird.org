Beyond that headline, we merged 347 PRs from 34 contributors, with major improvements to text editing, site-specific performance, CSS features, and SVG rendering.

### Adopting Rust for LibJS

Back in 2024, we tried adopting Swift as a second language alongside C++. It didn't pan out for us, so we removed all Swift code from the codebase ([#7999](https://github.com/LadybirdBrowser/ladybird/pull/7999)).

In its place, we landed a complete Rust reimplementation of the LibJS frontend pipeline: lexer, parser, AST, scope collector, and bytecode code generator ([#8104](https://github.com/LadybirdBrowser/ladybird/pull/8104)). The translation was done using AI coding agents (Claude Code and Codex), human-directed rather than autonomous. The Rust pipeline produces byte-for-byte identical bytecode output to the C++ implementation, verified across 52,898 test262 tests and 12,461 Ladybird-specific tests with zero regressions.

It's enabled by default on all platforms. You can switch back to the C++ pipeline by setting `LIBJS_CPP=1`, or run both pipelines in lockstep with `LIBJS_COMPARE_PIPELINES=1` to verify identical output.

For the full story on why we made this change and how the translation worked, see our [blog post about adopting Rust](/posts/adopting-rust/).

### Text editing and selection

We spent much of February overhauling text editing and selection, and the results are really noticeable. A bunch of interactions that felt broken or missing now just work:

- **Double-click and triple-click dragging**: You can double-click to select a word and drag to extend the selection word-by-word, or triple-click and drag to extend paragraph-by-paragraph ([#7771](https://github.com/LadybirdBrowser/ladybird/pull/7771))
- **Auto-scrolling**: Dragging a selection past the edge of a scrollable container now automatically scrolls to follow your cursor ([#7839](https://github.com/LadybirdBrowser/ladybird/pull/7839))
- **Scrollable text inputs**: When text in an `<input>` overflows the visible area, the content now scrolls properly instead of getting clipped
- **Caret rendering**: Empty `contenteditable` elements, `<input>`s, and `<textarea>`s now show a blinking caret where you'd expect one ([#7859](https://github.com/LadybirdBrowser/ladybird/pull/7859))
- **`::selection` styling**: CSS `::selection` rules now apply to text and form controls, including `text-decoration` and `text-shadow`
- **Tab key navigation**: Tabbing into an `<input>` now selects its contents, matching other browsers

### Performance

We continued profiling and optimizing specific popular websites:

- **Reddit**: Layout and rendering optimizations driven by profiling Reddit's infinite scroll ([#7872](https://github.com/LadybirdBrowser/ladybird/pull/7872))
- **YouTube**: Several rounds of profiling led to CSS property computation shortcuts, display caching, and eliminating unnecessary synchronous layouts ([#8051](https://github.com/LadybirdBrowser/ladybird/pull/8051), [#8059](https://github.com/LadybirdBrowser/ladybird/pull/8059), [#8084](https://github.com/LadybirdBrowser/ladybird/pull/8084))
- **YouTube Music**: Further layout and style optimizations ([#8076](https://github.com/LadybirdBrowser/ladybird/pull/8076))
- **slither.io**: IDL binding and overload resolution optimizations found by profiling the popular browser game ([#8203](https://github.com/LadybirdBrowser/ladybird/pull/8203))

There were also broader improvements:

- CSS custom properties now use structural sharing, so elements that inherit the same values share a single copy instead of each getting their own ([#7928](https://github.com/LadybirdBrowser/ladybird/pull/7928))
- Animated images (GIFs, etc.) now decode frames on demand instead of all at once, which cut over 1 GiB of memory usage on [cloudflare.com](https://www.cloudflare.com) alone ([#7930](https://github.com/LadybirdBrowser/ladybird/pull/7930))

### SVG partial relayout

Previously, any change to an SVG element's attributes triggered a full layout tree rebuild for the entire SVG. This month we landed partial SVG relayout ([#7816](https://github.com/LadybirdBrowser/ladybird/pull/7816)), so changes to transforms, viewBox, and other attributes only invalidate the affected subtree. This makes a big difference on sites like [Duolingo](https://www.duolingo.com) and [Cloudflare](https://www.cloudflare.com) that use animated SVGs extensively.

We also implemented `SVGPatternElement` ([#8011](https://github.com/LadybirdBrowser/ladybird/pull/8011)), the `feTurbulence` filter ([#7864](https://github.com/LadybirdBrowser/ladybird/pull/7864)), and `dominant-baseline` for SVG text ([#8173](https://github.com/LadybirdBrowser/ladybird/pull/8173)).

### Cookie versioning

The `document.cookie` property is used to read and write cookies from JavaScript. This property is synchronous, meaning
every cookie access blocks JavaScript execution while the browser fetches the latest cookie string. In modern multi-process
browsers, this requires inter-process communication to the process that stores the cookies. On real-world pages, scripts
frequently read document.cookie hundreds of times. The Chrome team [performed a study](https://blog.chromium.org/2024/06/introducing-shared-memory-versioning-to.html)
and found that 87% of these result in the same cookie string being returned.

Ladybird now implements cookie versioning and caching to eliminate most of those redundant look ups ([#7717](https://github.com/LadybirdBrowser/ladybird/pull/7717)).
A shared version counter tracks when cookies change, and the WebContent process keeps a cached copy of the cookie string.
If JavaScript calls `document.cookie` and the version hasn't changed, the cached value is returned immediately with no IPC.
As a result, hundreds of blocking calls during page load are eliminated.

We also rearchitected how we attach cookies the the HTTP `Set-Cookie` request header ([#7847](https://github.com/LadybirdBrowser/ladybird/pull/7847)).
We previously attached this header from the WebContent process within our Fetch implementation. This used the same cookie
infrastructure as `document.cookie`, so this access also blocked the page for every request. We moved this operation to
our networking process, where we can asynchronously attach the cookie header.

### Other notable changes

- **Fullscreen API**: We completed the spec-compliant implementation of `requestFullscreen()` and `exitFullscreen()`, including fullscreen event handlers, default UA styles, and proper cleanup when elements are removed or documents unloaded ([#7649](https://github.com/LadybirdBrowser/ladybird/pull/7649))
- **Video fullscreen**: With the Fullscreen API implemented, fullscreen could be easily implemented via the new shadow DOM controls ([#8129](https://github.com/LadybirdBrowser/ladybird/pull/8219))
- **LibJS architecture**: Scope analysis was separated from parsing ([#7852](https://github.com/LadybirdBrowser/ladybird/pull/7852)), the AST is now freed after bytecode generation ([#7896](https://github.com/LadybirdBrowser/ladybird/pull/7896)), and the runtime environment stack was replaced with explicit completion records ([#7840](https://github.com/LadybirdBrowser/ladybird/pull/7840), [#7846](https://github.com/LadybirdBrowser/ladybird/pull/7846))
- **JS `Number` fixes**: Improved accuracy of `Number.prototype.toPrecision` and `Number.prototype.toExponential` for extreme values ([#8036](https://github.com/LadybirdBrowser/ladybird/pull/8036), [#8063](https://github.com/LadybirdBrowser/ladybird/pull/8063))
- **JS `Iterator` features**: Implemented `Iterator.zip` and `Iterator.zipKeyed` ([#8191](https://github.com/LadybirdBrowser/ladybird/pull/8191))
- **Tab close confirmation**: Tabs with unsaved state can now prevent accidental closure via `beforeunload` ([#6743](https://github.com/LadybirdBrowser/ladybird/pull/6743))
- **Download images**: Right-click any image and download it directly ([#7735](https://github.com/LadybirdBrowser/ladybird/pull/7735))
- **test262 on PRs**: JavaScript conformance tests now run on every PR ([#8018](https://github.com/LadybirdBrowser/ladybird/pull/8018))
